## CHAPTER 1 — What an AI Playbook actually is

**[Statement block]** *Learning objective:* Describe what an AI Playbook does, where it runs, and what it produces.

**[Text block]**
An AI Playbook is an automated review that runs on Piai — EvenUp's PI-trained models — directly inside your case-management system. The moment a new file is added in Litify, SmartAdvocate, CASEpeer, or wherever your firm works, the Playbook reads it, extracts the key facts, flags risks and treatment gaps, and identifies high-value opportunities — and pushes those insights back into your CMS. No manual upload, no "remember to run it."

**[Flashcard grid block]** — the four things to know
- **Runs automatically** → triggered the moment a new case file lands. One-time configuration, then set-and-forget.
- **Reads every case** → the whole caseload gets the same first review, not just the file someone happened to open.
- **Cites every finding** → every output traces to the source with line-level citations you can verify in seconds.
- **Keeps a record** → a clear run history shows when each Playbook ran and what it generated.

**[Text block]**
The point isn't to replace your read — it's to make sure *every* file gets a consistent first read, so nothing waits for a human to get to it before a risk or a deadline surfaces.

## CHAPTER 2 — Configure your first Playbook

**[Statement block]** *Learning objective:* Set up a focused first Playbook that reflects the firm's best practices — and start narrow.

**[Text block]**
A Playbook is only as useful as what you tell it to look for. The "one-time configuration" is where the firm's standards get baked in: which case types to cover, which facts to extract, and which risks to flag. The temptation is to turn on everything at once. Don't. Start with one case type and a couple of high-value checks, prove it, then expand.

**[Process block]** — configuring the pilot Playbook
1. **Pick the trigger.** Scope it to one case type for the pilot — e.g., motor-vehicle matters — so results are easy to judge.
2. **Choose what to extract.** The key facts the team uses on every file: injuries, providers, treatment dates, billed amounts.
3. **Choose what to flag.** Start with the highest-value signals: treatment gaps, missing records, and pre-existing conditions.
4. **Apply firm best practices.** Encode the firm's SOPs — what "complete" looks like, what always gets escalated.
5. **Confirm it writes back.** Check that insights push into the CMS where the team already works.

**[Text block]**
At Lakefront Injury Group, the pilot Playbook covers standard motor-vehicle matters and flags three things: treatment gaps over 30 days, missing provider records, and any noted prior injury. Narrow, high-value, easy to trust — then expand once it's earning its keep.

**[Knowledge check — scenario]**

## CHAPTER 3 — Trust it: verify in seconds with line-level citations

**[Statement block]** *Learning objective:* Use line-level citations and the run history to validate Playbook output quickly — and know what to spot-check.

**[Text block]**
Proactive only works if you trust what the Playbook surfaces. EvenUp builds that in: every fact it extracts and every flag it raises traces to the exact line in the source document. You don't re-read the file — you verify the high-stakes items at their citation, in seconds. This is the same review-against-source habit that earns attorney trust, made fast.

**[Accordion or numbered list block]** — the verification pass
1. **Start with the flags.** Treatment gaps, missing records, pre-existing conditions — the items that change case strategy.
2. **Click the citation.** Each one lands on the source line that supports it. If the line says it, trust it and move on.
3. **Spot-check the dollars and dates.** Billed amounts and key treatment dates drive valuation — verify these even when they look right.
4. **If the line doesn't support it, fix or flag it.** A citation that doesn't check out is exactly what you're here to catch.
5. **Use the run history.** A clear record of when the Playbook ran and what it produced — your audit trail and your check that nothing was missed.

**[Statement block]**
> Line-level citations turn "do I trust the AI?" into a five-second click. Verify the high-stakes items; let the citation carry the rest.

**[Text block]** — a note on security
This runs on a platform built for PI work — SOC 2 and HIPAA standards, enterprise controls — not a general-purpose chatbot. That's what makes it safe to put protected health information and client records through it in the first place. Trust in the output and protection of the data are both part of why the Playbook can run on every file.

**[Knowledge check — multiple response]**

- Click the line-level citation on each high-stakes flag ✅
- Spot-check billed amounts and key treatment dates ✅

- Use the run history to confirm what ran and when ✅
- Fix or flag any citation that doesn't support the claim ✅
Feedback: Verify smartly at the highest-stakes points using the citations — don't redo the work the Playbook just did.

## CHAPTER 4 — Act on it, then make it better

**[Statement block]** *Learning objective:* Turn Playbook findings into case action, and refine the Playbook so it improves over time.

**[Text block]**
A Playbook's output is a to-do list for the case, already sorted. Every flag becomes an action with an owner — and because the insights live in your CMS, the next step happens where the team already works.

**[Numbered list block]** — from finding to action
1. **Treatment gap** → open client/provider follow-up before it costs case value.
2. **Missing record** → trigger a records request; the run history tracks that it was caught.
3. **Pre-existing condition / causation question** → escalate to the attorney; it can move case theory and damages.
4. **High-value opportunity / key facts** → feed straight into the demand via AI Drafts.
5. **Nothing flagged** → that's a verified clean first read, logged — not a file that simply never got looked at.

**[Text block]** — make it better at scale
Set-and-forget isn't set-and-ignore. Review the run history, notice what the team keeps correcting, and tune the Playbook — adjust a flag, tighten a standard, add a check. Save what works so the whole firm's Playbooks apply the same best practices at scale. That feedback loop is how a good Playbook becomes the firm's institutional knowledge, running on every case.

**[Knowledge check — scenario]**

- C) Turn off the pre-existing-condition flag so it stops surfacing
- D) Wait for the attorney to find it
Feedback (correct): Verify, then route. The Playbook surfaced it; your judgment and action are what turn a flag into case value.

### One-page job aid (attach in the Close)

**Your First AI Playbook — configure · trust · act · refine**

1. **Configure:** one case type, 2–3 high-value flags (treatment gaps, missing records, pre-existing conditions), firm SOPs baked in, writes back to the CMS.
2. **Trust:** verify high-stakes flags at their line-level citation; spot-check dollars and dates; use the run history.
3. **Act:** gap → follow-up · missing record → request · pre-existing/causation → escalate · key facts → feed the demand.
4. **Refine:** watch what the team keeps correcting, tune the flags, save what works for the whole firm.

*The rule of thumb: if a flag changes what someone does next, verify it at the citation, then route it. Set-and-forget, never set-and-ignore.*
